# Triage: 2026-04-08 LanceDB Bloat Spiral & Stale Daemon

## Summary

Three consecutive patch releases (v0.14.7, v0.14.8, v0.14.9) attempted to fix daemon zombie accumulation and resource waste. The zombie issue is now fixed, but the underlying LanceDB bloat problem has worsened — the vector store hit **350GB peak** (26.9x bloat) and is still at **13GB** (only just compacted from 93GB during this investigation). A stale daemon process (PID 7227, 4 hours old) was found running alongside the active daemon despite the v0.14.8/v0.14.9 fixes. The root cause is a self-reinforcing feedback loop between FSEvents overflow, unchanged file re-processing, and compaction conflicts.

## Root cause: the bloat spiral

This is a self-reinforcing loop. Each stage makes the next one worse:

```
FSEvents overflow on "platform" (402K chunks, large project)
    |
    v
Full re-scan triggered (catchup queues all changed files)
    |
    v
Batch processor sends thousands of .graphql.swift files to workers
  (mtime changed by Xcode codegen, content identical)
  (batch processor lacks the fast in-process hash check that catchup has)
    |
    v
Each file: delete old vectors + insert new = 2 LanceDB version bumps
  Even "0 reindexed" files create version churn from the delete check
    |
    v
Fragments accumulate (each batch of 50 files = new fragments)
    |
    v
Scheduled compaction (every 5min) tries to merge fragments
  BUT: conflicts with concurrent batch processor deletes/inserts
  Retries 5 times with exponential backoff (1s, 2s, 4s, 8s, 16s)
  Often fails all 5 attempts
    |
    v
Disk bloat grows (93GB for 10.6GB logical = 8.7x)
  More I/O pressure from bloated reads
    |
    v
FSEvents buffer fills faster -> more drops -> loop repeats
```

### Why compaction keeps losing

The batch processor and compaction run concurrently with no coordination. LanceDB uses optimistic concurrency — a `Rewrite` transaction (compaction) is preempted by any concurrent `Delete` transaction (batch processor cleanup). Since the batch processor runs continuously during catchup, compaction almost never gets a clean window:

```
[vectordb] Optimize conflict (attempt 1/5), retrying in 1000ms
[vectordb] Optimize conflict (attempt 2/5), retrying in 2000ms
[vectordb] Optimize conflict (attempt 3/5), retrying in 4000ms
[vectordb] Optimize conflict (attempt 4/5), retrying in 8000ms
[vectordb] Optimize failed: lance error: Retryable commit conflict for version 1155:
  This Rewrite transaction was preempted by concurrent transaction Delete
```

This is the core issue. The v0.14.7-v0.14.9 releases fixed daemon startup races but never touched the compaction-vs-writes conflict.

## What the last 3 releases fixed (and didn't)

### v0.14.7 — Watcher ignore patterns
Added missing ignore patterns (`.venv`, `site-packages`, `dist`, `build`, `target`, `vendor`, etc.) to both the FSEvents watcher and batch processor regex. Reduced unnecessary indexing work but did not address bloat or zombies.

### v0.14.8 — Zombie exit on lock compromise
When `proper-lockfile` fires `onCompromised`, the old daemon now calls `shutdown().finally(() => process.exit(0))` with a 10-second kill timer. Fixed the symptom (zombie hangs forever) but not the cause (why locks get compromised).

### v0.14.9 — Startup order fix
Reordered daemon startup: socket server now listens **before** PID file is written. Eliminates the TOCTOU race where a new daemon sees the PID, pings the socket (not listening yet), concludes the daemon is dead, and kills it. This is the real zombie fix.

### What none of them fixed
- LanceDB bloat spiral (compaction vs concurrent writes)
- Batch processor missing fast hash check (Known Issue #3 from CLAUDE.md)
- FSEvents overflow -> re-scan -> wasted work cycle
- Fragment accumulation during sustained writes

## Stale daemon (PID 7227)

Despite v0.14.8 and v0.14.9 fixes, PID 7227 survived as a stale daemon for 4 hours. This daemon was started at 6:36 PM; the active daemon (PID 37894) started at 7:49 PM.

**Why it survived:** The v0.14.9 startup reorder only prevents the *new* daemon from killing a healthy old daemon during the startup race. If the old daemon (7227) was started before v0.14.9 was deployed (i.e., running old code), it wouldn't have the `process.exit()` on lock compromise from v0.14.8 either. The stale daemon is a leftover from pre-fix code.

**Remediation:** Killed PID 7227 during this investigation. This is not a recurring bug — it's a one-time artifact of deploying the fix while the old daemon was running.

## Uncommitted changes (in-progress fixes)

The working tree has changes across 7 files that form a cohesive **disk pressure protection and fragment prevention system**:

### 1. Disk pressure detection (`vector-db.ts`)
- `DiskPressureLevel` type: `"ok" | "low" | "critical"`
- `checkDiskPressure()`: cached check (30s interval) using `statfsSync()`
- `ensureDiskOk()`: guard on all LanceDB writes, throws `DiskPressureError` at critical
- `optimize()` ENOSPC handling: stops retrying on "No space left" instead of making it worse
- `runMaintenance()`: skips at critical, single-pass at low, full optimize+bloat at ok
- `compactIfNeeded(threshold)`: on-demand compaction, guards on `maintenanceRunning`

### 2. Batch processor circuit breaker (`batch-processor.ts`)
- Checks `diskPressure` before processing; defers 60s at critical
- Catches `DiskPressureError`, requeues files without counting retries
- Fires `compactIfNeeded()` after every batch with reindexed files

### 3. Syncer flush batches (`syncer.ts`)
- `batchLimit` changed from 24 to 2000 (reduces fragments from ~834 to ~5 for 10K chunks)
- Mid-sync compaction every 10 flushes via `compactIfNeeded(30)`

### 4. Catchup throttling (`daemon.ts`)
- Pauses 5s every 500 queued files during catchup scan
- `getDiskPressure()` method for IPC
- `getPortPid()` + stale MLX process cleanup before spawning

### 5. IPC status (`ipc-handler.ts`)
- Exposes `diskPressure` in status response

### 6. Doctor disk check (`doctor.ts`)
- Reports disk space and pressure level
- Uses config constants for thresholds (fixed during this session — was using hardcoded values)

### 7. Config constants (`config.ts`)
- `DISK_CRITICAL_BYTES` (5GB, env-overridable)
- `DISK_LOW_BYTES` (20GB, env-overridable)
- `FRAGMENT_COMPACT_THRESHOLD` (50)

## What these changes fix and what they don't

### Fixed by uncommitted changes
- **ENOSPC spiral**: disk pressure detection prevents writes when disk is critically low
- **Fragment accumulation during initial sync**: 2000-record flush batches (was 24)
- **Fragment accumulation during live watch**: post-batch `compactIfNeeded()`
- **Catchup flooding**: 5s throttle every 500 files gives compaction windows
- **MLX orphan cleanup**: stale processes on port 8100 killed before spawn
- **Observability**: disk pressure in `gmax status` and `gmax doctor`

### NOT fixed (remaining issues)

1. **Compaction still loses to concurrent writes.** The fundamental conflict between batch processor deletes and compaction Rewrite transactions is unchanged. `compactIfNeeded()` guards on `maintenanceRunning` to prevent overlapping compaction, but doesn't coordinate with batch processor writes. The batch processor should **pause writes** during compaction, or compaction should only run between batches.

2. **Batch processor still lacks fast hash check (CLAUDE.md Known Issue #3).** The catchup scan has the in-process hash shortcut (`daemon.ts:450-461`) but the batch processor (`batch-processor.ts:157-179`) still does the full worker round-trip for every file. For the "platform" project with thousands of mtime-only changes from Xcode codegen, this means ~220ms/file wasted on worker dispatch for files that are 100% unchanged. This is the single biggest source of wasted work.

3. **FSEvents overflow -> re-scan loop.** When FSEvents drops events, the watcher triggers a full re-scan. With 402K chunks in "platform", this is extremely expensive and the most common trigger for the bloat spiral. No backpressure or debouncing on re-scans.

4. **No compaction window during sustained writes.** The batch processor debounces at 2s and processes batches of 50. During catchup of thousands of files, there's never a gap long enough for compaction to succeed. The 5s throttle pause every 500 files helps but doesn't guarantee a clean window — the batch processor may still be flushing its current batch during the pause.

5. **`daemon.log.prev` at 71MB.** Log rotation target is 5MB. Either rotation failed or the log grew extremely fast in one session. The log should be examined and cleaned up.

## Recommended fixes (priority order)

### P0: Coordinate compaction with writes

The batch processor should gate writes during compaction. Simplest approach: when `compactIfNeeded()` returns true (compaction needed), the batch processor should wait for it to complete before starting the next batch. This gives compaction a clean transaction window.

Alternatively, run compaction in the batch processor's `finally` block after all writes are done, instead of on the 5-minute maintenance timer.

### P1: Add fast hash check to batch processor

Port the catchup scan's in-process hash shortcut to `batch-processor.ts`:
```typescript
// In processBatch(), after stat check:
const cached = this.metaCache.get(absPath);
if (cached && cached.hash && cached.size === stats.size) {
  const { computeBufferHash } = await import("../utils/file-utils");
  const buf = await fs.promises.readFile(absPath);
  const hash = computeBufferHash(buf);
  if (hash === cached.hash) {
    this.metaCache.put(absPath, { ...cached, mtimeMs: stats.mtimeMs });
    continue; // Skip worker round-trip
  }
}
```

This eliminates the primary source of wasted work (thousands of .graphql.swift files sent to workers for nothing).

### P2: Debounce FSEvents overflow re-scans

When FSEvents drops events, don't immediately re-scan the entire project. Instead:
- Debounce re-scans to at most once per 30 seconds
- During re-scan, only check files modified since the last successful scan (use MetaCache mtime as baseline)
- Log overflow events to track frequency

### P3: Clean up log rotation

Investigate why `daemon.log.prev` grew to 71MB. Either the rotation trigger isn't checking the `.prev` file size, or the daemon generated 71MB of logs in a single rotation window (plausible during the bloat spiral — 129 conflict logs + 35 bloat logs + batch processing of thousands of files).
