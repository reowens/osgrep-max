# Triage: 2026-04-03 Runaway Processes

## Summary

On the morning of 2026-04-03, the gmax daemon was found in a catastrophic state: two daemon instances running simultaneously, 7 workers consuming 500%+ CPU with a frozen log, the MLX embed server using 26GB of Metal GPU memory, and the LLM summarizer making requests despite being "disabled." The daemon log had stopped growing at 587 lines — there was no way to diagnose what was happening because debug logging didn't exist in any of the hot paths.

Root cause: the daemon was never designed as a full process manager. It was built as a write coordinator (v0.13.0) and never gained responsibility for the MLX embed server, error recovery, or operational observability. Every debugging session rediscovers the same gaps because there was no architecture document.

---

## Bugs Found

### Bug 1: Two daemons running simultaneously

**Symptoms:** PIDs 61294 (idle, 2h50m old) and 50530 (80% CPU, 5min old) both running as `gmax-daemon`. Old daemon never killed.

**Root cause:** Lock staleness was set to 30 seconds (`daemon.ts:55`, `stale: 30_000`). The daemon heartbeats every 60s. A slow startup or heavy indexing could leave the lock untouched for >30s, causing `proper-lockfile` to consider it stale and letting a second daemon acquire it. The old daemon's socket was cleaned up (step 4 in startup), so `isDaemonRunning()` returned false even though the process was alive.

**Fix applied:** 
- Added PID file validation before lock acquisition: read PID -> check alive -> socket ping -> kill if unresponsive (`daemon.ts:52-70`)
- Changed `stale: 30_000` to `stale: 120_000` (`daemon.ts:79`)

**Status:** FIXED (uncommitted)

---

### Bug 2: MLX embed server never started by daemon

**Symptoms:** Every embed batch logged `mlx=false`, falling back to ONNX CPU at ~5x slower. `curl http://127.0.0.1:8100/health` returned connection refused. The MLX server that had been running (PID 49430) was a leftover from a previous session, killed with the runaway processes.

**Root cause:** `startMlxServer()` only exists in `src/commands/serve.ts:45`. The daemon (`src/lib/daemon/daemon.ts`) has zero references to MLX, embed server, port 8100, or GPU. When the daemon took over file watching in v0.13.0, nobody moved MLX server management from `serve` to the daemon. Every project and the global config say `embedMode: "gpu"` but the daemon never acts on it.

**Fix applied:** None. This is an architecture gap, not a simple bug.

**Status:** OPEN. Priority #1. The daemon must start and manage the MLX embed server when `embedMode` is `"gpu"`.

### Bug 3: Catchup scan re-queuing thousands of unchanged files

**Symptoms:** Platform catchup queued 2027+ files on every daemon restart. Batch processor ground through them all at "0 reindexed." Remaining count jumped from 444 back to 2753 mid-processing because the daemon was killed and restarted, triggering another catchup of the same files.

**Root cause:** Something external (likely Xcode/code generation) was touching `.graphql.swift` files, changing their mtime without modifying content. The catchup scan (`daemon.ts:329-333`) compared mtime+size via `isFileCached()` — mtime differed, so every file was queued. The batch processor then sent each file to a worker (~220ms round-trip), got a hash match, updated meta, and reported "0 reindexed." On next restart, the files had been touched again, and the cycle repeated.

Debug logging added to catchup confirmed:
```
[catchup] miss ...graphql.swift: cached=mtime=1775216632354 stat=mtime=1775218400171 size=28416
```
Same size, different mtime, identical content.

**Fix applied:** Added fast in-process hash check in catchup scan (`daemon.ts:335-344`). When mtime changed but size matches and cached hash exists, reads file and computes hash in the daemon process. If hash matches, updates mtime in MetaCache and skips the worker entirely. Result: catchup went from 2027 queued to 21 queued (only genuinely new files with `cached=null`).

**Status:** FIXED (uncommitted). Note: the batch processor (`batch-processor.ts:142-161`) still has the slow path for files queued by the live watcher — it sends every mtime-changed file to a worker. The catchup fast-path should be applied there too.

### Bug 4: Workers stuck at high CPU, daemon log frozen

**Symptoms:** 7 workers at 60-84% CPU each, daemon at 157% CPU. Daemon log stuck at 587 lines, not growing. `sample` showed workers deep in V8 JS execution (promise chains, generators), not I/O-blocked. No way to tell what they were doing.

**Root cause:** Zero debug logging in the hot paths. The worker pool (`pool.ts`) had no logging on task dispatch, completion, error, or timeout. The worker child process (`process-child.ts`) had zero comments and zero logging. The orchestrator (`orchestrator.ts`) had no logging on file processing, chunking, or embedding. The MLX client (`mlx-client.ts`) had no timing on HTTP requests.

**Fix applied:** Added comprehensive debug logging gated behind `GMAX_DEBUG=1` across 8 files:
- `logger.ts` — added `debugTimer()`, `debugEvery()`, exported `VERBOSE`
- `pool.ts` — dispatch, complete, error, timeout, spawn, reap, queue depth
- `process-child.ts` — task recv/done/fail with timing
- `orchestrator.ts` — processFile start/done, chunk count, embed batch timing, MLX indicator
- `mlx-client.ts` — HTTP timing, health checks, cache state
- `syncer.ts` — walk progress, file outcomes, flush stats
- `daemon.ts` — lock, socket, IPC commands, catchup stats
- `ipc-handler.ts` — command logging

**Status:** FIXED (uncommitted)

### Bug 5: MLX embed server using 26GB Metal memory

**Symptoms:** User observed `gmax-embed` (PID 49430) consuming 26GB. The process was the MLX embedding server running on Metal GPU.

**Root cause:** `mlx-embed-server/server.py` `embed_texts()` function created intermediate MLX tensors (`input_ids`, `attention_mask`, `outputs`, `pooled`, `norms`) but never freed them or cleared the Metal cache. The summarizer (`summarizer.py:151`) correctly called `mx.metal.clear_cache()` but the embed endpoint did not. Over hundreds of embedding requests during a large re-index, Metal GPU memory accumulated without bound.

**Fix applied:** Added `del input_ids, attention_mask, outputs, pooled, norms` and `mx.metal.clear_cache()` after every embed call (`server.py:95-96`). Added debug logging with Metal memory stats when `GMAX_DEBUG=1` is set (`server.py:150-157`). Added periodic memory stats in idle watchdog (`server.py:112-118`).

**Status:** FIXED (uncommitted)

### Bug 6: 7 workers pre-allocated when 1 is needed

**Symptoms:** 7 `gmax-worker` processes (87MB each, ~525MB total) sitting idle while only 1 was doing work. The batch processor serializes file processing (one at a time per project), so during catchup only 1 worker was busy.

**Root cause:** `WorkerPool` constructor (`pool.ts:137-139`) eagerly spawned `CONFIG.WORKER_THREADS` workers (7 on a 14-core machine) on initialization. No idle reaping existed.

**Fix applied:**
- Changed constructor to spawn 1 worker on init, scale up on demand in `dispatch()` (`pool.ts:144`)
- Added `reapIdleWorkers()` on 60s interval — kills workers idle >60s, always keeps at least 1 (`pool.ts:476-498`)
- Added `lastBusyTime` tracking on `ProcessWorker` (`pool.ts:87`)
- Changed `handleWorkerExit` to only respawn if workers=0 or pending tasks exist (`pool.ts:209-218`)

**Status:** FIXED (uncommitted)

### Bug 7: `tools` project stuck in error status

**Symptoms:** `projects.json` shows `tools` project with `status: "error"`, `chunkCount: 0`, `lastIndexed: ""`. The daemon ignores it on every startup.

**Root cause:** The `tools` project was added as a consolidated root (`/Users/reoiv/Development/beyond/tools`) replacing individual sub-projects (gmax, cc-src, sentinel, dotmd, music-ripper). The runaway daemon (PID 50530) attempted `indexPendingProject` for it, found 34% coherence (2362 vectors vs 6998 cached), cleared the cache, and started re-embedding ~7000 files. This daemon was eventually killed, and `indexPendingProject` caught the error and presumably left the status as `"error"` (or it was set during a failed `gmax add`).

The daemon filters projects on startup:
- `status === "indexed"` -> watch (`daemon.ts:126`)
- `status === "pending"` -> background index (`daemon.ts:140`)
- `status === "error"` -> **ignored**

No retry mechanism exists. No background recovery. No alerting.

**Status:** OPEN. The project must be manually re-added (`gmax add /path/to/tools`) or `projects.json` edited to change status to `"pending"`.

### Bug 8: LLM summarizer called despite being "disabled"

**Symptoms:** `llm-server.log` showed 15 `POST /v1/chat/completions` requests. The LLM server (llama-server) loaded a ~21GB model into Metal GPU memory. User stated summarization is supposed to be disabled.

**Root cause:** Not fully diagnosed. The LLM server log shows it was running and processing requests. The daemon constructs `LlmServer` on startup (`daemon.ts:119`) but doesn't start it — it's lazy, started on first `llm-start` IPC command or `reviewCommit()`. Something triggered it. Possibilities:
1. An MCP tool call triggered `summarize_directory` or `summarize_project` (`mcp.ts:2168-2172`)
2. A commit review was triggered via hook or IPC
3. The `gmax llm start` command was run manually
4. The `ensure()` method was called by investigate/review code paths

The LLM server exited before the session ended (log shows `ggml_metal_free: deallocating`). The 15 requests suggest a short automated burst, not manual usage.

**Status:** UNEXPLAINED. Need to check: is there a SessionStart hook or git hook that triggers `gmax llm start` or `gmax review`? The `generateSummaries` function (`syncer.ts:44`) does a test call to the summarizer before proceeding — if the LLM was already running, this would succeed and trigger summarization during indexing.

## Systemic Issues: Why This Keeps Happening

### No architecture document existed

README.md line 296 references a `CLAUDE.md` that didn't exist. There was no document explaining what processes gmax runs, who starts them, what the daemon manages, or what it doesn't. NEXT.md documents the v0.13.0 design decisions but not the current state.

### Daemon was built as a write coordinator, not a process manager

NEXT.md Part 1 describes the daemon's purpose: eliminate write contention by routing all LanceDB writes through one process. It succeeded at that. But it never took on the broader role of managing the full embedding pipeline — the MLX server, worker health, error recovery, or operational observability. Those responsibilities stayed in `serve.ts` (which nobody runs as a daemon) or were never implemented.

### 3.1% comment ratio, zero TODO markers

Across the 7 critical daemon/indexing/worker files (3,528 lines), there are ~124 comment lines. No TODO, FIXME, HACK, XXX, or BUG markers anywhere. Complex functions like `processBatch()` (270 lines), `processFile()` (75 lines), and `initialSync()` (400+ lines) have minimal documentation. The numbered startup steps in `daemon.ts` have a duplicate label error (two `// 7.` entries). Known gaps aren't tracked in the code.

### No debug logging in production paths

Before this session, `GMAX_DEBUG=1` controlled only the `debug()` calls in `logger.ts`, which were used in exactly 2 places: `syncer.ts` (SKIP/EMBED per file) and `pool.ts` (worker spawn). The entire worker lifecycle, embedding pipeline, MLX client, batch processor, and daemon catchup had zero debug instrumentation. When workers hung, the only diagnostic was `ps aux` and `sample`.

### No error recovery

Projects that fail indexing enter `"error"` status and are permanently ignored. Workers that crash 10+ times in a row cause the pool to stop spawning, but the batch processor doesn't detect this. The LLM server can die and leave a stale PID file. The MLX server isn't managed at all. There's no retry, no alerting, no self-healing.
